## Code of Conduct

We expect all contributors to treat each other with respect. Harassment, discrimination, and bad-faith participation of any kind will not be tolerated.

## Reporting Bugs

Open an issue and include:

- A clear, descriptive title
- Step-by-step reproduction instructions
- Expected behaviour vs. actual behaviour
- Your environment (OS, Python version, dependency versions)

---

## Suggesting Features

Open an issue and include:

- The problem you are trying to solve
- Your proposed solution
- Alternatives you considered

## Submitting Changes

1. Fork the repository
2. Branch from `main`
3. Make your change
4. Commit following the style below
5. Open a pull request against `main` and describe *why* the change is needed

One concern per PR — do not bundle unrelated changes.

---

## Commit Style

Commits use a bracketed title line followed by a Conventional Commits body.

**Bracket table:**

| Bracket | Meaning |
|---------|---------|
| `[A]` | Add — new feature or file |
| `[C]` | Change — behaviour change to existing feature |
| `[D]` | Deprecate — mark something for future removal |
| `[R]` | Remove — delete a feature or file |
| `[F]` | Fix — bug fix |
| `[S]` | Security — security patch |
| `[B]` | Build / dependency change |
| `[CI]` | CI/CD pipeline change |
| `[T]` | Test — add or update tests |
| `[DOC]` | Documentation only |
| `[I18N]` | Internationalisation / localisation |
| `[REF]` | Refactor — no behaviour change |
| `[PERF]` | Performance improvement |
| `[WIP]` | Work in progress (not ready to merge) |

**Rules:**

- Title line: `[BRACKET] Short imperative description` (≤ 72 chars)
- Body: optional, use Conventional Commits style (`feat:`, `fix:`, etc.)
- Reference issues with `Closes #N` or `Refs #N` in the body
- No trailing period on the title
